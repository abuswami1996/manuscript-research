# Protocol Browser Frontend — Analysis & Recommendation

## 1. Current State Assessment

### What the ideation agent produces

Each completed run lives at `workspace/runs/<project>/<timestamp>/` and contains:

| File | Purpose | Size (example run) |
|---|---|---|
| `output/study_protocol.md` | The deliverable — a complete, IRB-ready study protocol | ~900 lines |
| `output/protocol_metadata.json` | Structured metadata (generated by `extract_metadata.py`) | ~260 lines |
| `survey/landscape_report.md` | Synthesized gaps and opportunities | ~450 lines |
| `survey/raw_sources.md` | Retrieved abstracts and source metadata | ~675 lines |
| `ideas/candidate_ideas.md` | 5-10 study concepts with hypotheses and methods | ~570 lines |
| `critiques/critique_report.md` | Evaluation with scores, objections, prior art | ~480 lines |
| `critiques/novelty_checks.md` | Raw API search results for each candidate | variable |
| `scratchpad/principal_notes.md` | Principal's reasoning, selection rationale | ~70 lines |
| `scratchpad/critic_notes.md` | Critic's methodology notes | variable |

There are currently 4 ideation runs with completed protocols (plus 1 literature search run). Another run is in progress.

### What's already good for a frontend

1. **`protocol_metadata.json` is frontend-ready.** It was designed for this purpose. Controlled vocabularies (`Specialty`, `StudyType`) via `Literal` types mean filterable facets without normalization work. Topic slugs are canonicalized. The `files` field maps logical names to relative paths, providing a table of contents.

2. **References have DOIs and PMIDs.** 15 references per protocol with structured `doi`, `pmid`, `role` (background / gap-evidence / prior-art / methodology). DOIs link to `https://doi.org/{doi}`, PMIDs to `https://pubmed.ncbi.nlm.nih.gov/{pmid}/`. Not all references have both identifiers, but enough for clickable links.

3. **Markdown is well-structured.** The protocol uses standard markdown — headers, tables, code blocks (for ASCII diagrams), numbered lists, bold/italic. `react-markdown` + `remark-gfm` handles all of it. Each section has a clear `## SECTION N:` header, enabling a generated table of contents.

4. **Ideation process metadata is captured.** `ideation.ideas_considered`, `ideation.critique_cycles`, `ideation.selected_idea`, `ideation.selection_rationale`, and `ideation.scores` (novelty/feasibility/impact) are all extracted. This gives the frontend a "how was this made?" story.

5. **The `files` field provides document navigation.** Each key in `files` maps to a workspace-relative path, so a tabbed document viewer can link directly to each supporting document.

### Gaps and things to address

#### A. Missing metadata fields (add to `extract_metadata.py`)

| Field | Why | Source |
|---|---|---|
| `rejected_ideas` | Users want to see "what else the agent considered" — it's interesting UX and demonstrates breadth | `candidate_ideas.md` + `critique_report.md` |
| `agent_model` | Transparency: which LLM produced this protocol | Runtime env var or new field |
| `user_prompt` | What prompt (if any) was given to the agent — adds context on intent vs. fully autonomous | Runtime or log |
| `total_references_in_protocol` | The metadata captures top 10-15, but the protocol has 30+. A count tells users "15 of 35 shown" | `study_protocol.md` references section |

`rejected_ideas` is the most impactful addition — a list of `{title, reason, scores}` for ideas the agent evaluated but didn't select. The critique report has this data; it just needs extraction.

#### B. Reference completeness

The metadata extracts 10-15 references, but the protocol cites 30+. Two options:

1. **Keep the metadata references as "key references" and parse the full reference list from the protocol's `## SECTION 15: REFERENCES` into a separate `all_references` field.** This is straightforward — the references section uses a consistent numbered format with DOIs and PMIDs inline.
2. **Just show 10-15 in the card view, and let users find all references in the full protocol text.** Simpler, and the full text is searchable anyway.

Recommendation: option 2 for v1. The protocol markdown already renders a complete references section. The metadata references power the "key references" sidebar on the detail page.

#### C. Markdown rendering considerations

- **ASCII diagrams**: The protocol contains text-art flow diagrams inside ` ``` ` code blocks. These render correctly in monospace with standard markdown rendering.
- **Long documents**: The protocol is ~900 lines. The detail page needs a sticky table of contents or section navigation. Parse `##` headings from the markdown to auto-generate this.
- **Tables**: Standard markdown tables are used throughout (budget, sample size, timeline). `remark-gfm` handles these.
- **Internal cross-references**: The principal notes reference other workspace files (e.g., "Read /survey/landscape_report.md"). These could be auto-linked to the corresponding tab, but this is a v2 enhancement.

#### D. File storage strategy

The workspace files are local markdown. For the frontend to serve them:

- **Upload markdown content to Supabase Storage** (or store as text in a `protocol_files` table). Storage is simpler for large text and avoids bloating the database.
- The hydration script reads each run directory, uploads files to storage at a path like `runs/{protocol_id}/{logical_name}.md`, and records the storage path in the database.

#### E. What the agent does NOT produce

- **No images or figures.** Protocols are text-only. Charts and diagrams are described in text or drawn in ASCII. This means the frontend is a reading/text experience — no image gallery needed.
- **No structured "alternative ideas" output.** The critique report has this information but it's embedded in free-text markdown, not structured. The `rejected_ideas` metadata field addresses this.
- **No agent execution trace.** There's no structured log of "surveyor ran for 3 minutes, then ideator ran for 2 minutes." This could come from LangSmith traces in the future but isn't needed for v1.

## 2. Data Model (Supabase)

### Tables

```sql
-- Core protocol record (one row per completed run)
create table protocols (
  id            uuid primary key,          -- = protocol_id from metadata
  run_id        text not null unique,      -- = directory timestamp name
  title         text not null,
  short_title   text not null,
  abstract      text not null,
  keywords      text[] not null,           -- postgres array
  specialty     text not null,             -- controlled vocabulary
  subspecialties text[] not null,
  study_type    text not null,             -- controlled vocabulary
  primary_outcome text not null,
  secondary_outcomes text[] not null,
  data_sources  text[] not null,
  estimated_budget_usd integer,
  status        text not null default 'complete',
  version       text not null default '1.0',
  created_at    timestamptz not null,
  inserted_at   timestamptz not null default now(),

  -- Denormalized ideation process (avoids a join for every card render)
  domain_chosen       text,
  ideas_considered    integer,
  critique_cycles     integer,
  selected_idea       text,
  selection_rationale text,
  score_novelty       integer,
  score_feasibility   integer,
  score_impact        integer
);

-- Full-text search index
create index protocols_fts on protocols
  using gin (to_tsvector('english',
    title || ' ' || short_title || ' ' || abstract || ' ' || array_to_string(keywords, ' ')
  ));

-- Topics (normalized — many protocols may share the same topic slug)
create table topics (
  id    text primary key,   -- lowercase-hyphenated slug
  label text not null
);

create table protocol_topics (
  protocol_id uuid references protocols(id) on delete cascade,
  topic_id    text references topics(id) on delete cascade,
  primary key (protocol_id, topic_id)
);

-- References
create table protocol_references (
  id          uuid primary key default gen_random_uuid(),
  protocol_id uuid not null references protocols(id) on delete cascade,
  title       text not null,
  authors     text not null,
  year        integer not null,
  journal     text not null,
  doi         text,
  pmid        text,
  role        text not null,   -- background | gap-evidence | prior-art | methodology
  sort_order  integer not null default 0
);

-- Study design (1:1 with protocol, separate for cleanliness)
create table study_designs (
  protocol_id              uuid primary key references protocols(id) on delete cascade,
  description              text not null,
  is_multicenter           boolean not null,
  has_control_arm          boolean not null,
  has_randomization        boolean not null,
  estimated_duration_months integer,
  target_sample_size       integer
);

-- Workspace files — pointers to Supabase Storage objects
create table protocol_files (
  id          uuid primary key default gen_random_uuid(),
  protocol_id uuid not null references protocols(id) on delete cascade,
  logical_name text not null,           -- e.g. "protocol", "landscape_report"
  storage_path text not null,           -- Supabase Storage path
  display_name text not null,           -- e.g. "Study Protocol", "Landscape Report"
  unique (protocol_id, logical_name)
);

-- Votes (authenticated users only)
create table votes (
  user_id     text not null,            -- Clerk user ID
  protocol_id uuid not null references protocols(id) on delete cascade,
  value       smallint not null check (value in (-1, 1)),
  created_at  timestamptz not null default now(),
  primary key (user_id, protocol_id)
);

-- Comments (authenticated users only)
create table comments (
  id          uuid primary key default gen_random_uuid(),
  protocol_id uuid not null references protocols(id) on delete cascade,
  user_id     text not null,            -- Clerk user ID
  body        text not null,
  created_at  timestamptz not null default now(),
  updated_at  timestamptz not null default now()
);
create index comments_protocol on comments(protocol_id, created_at);
```

### Row-Level Security

```sql
-- Protocols, topics, references, files: public read
alter table protocols enable row level security;
create policy "Public read" on protocols for select using (true);

-- Votes: public read, authenticated insert/update/delete own
alter table votes enable row level security;
create policy "Public read" on votes for select using (true);
create policy "Auth write own" on votes for all
  using (auth.uid()::text = user_id)
  with check (auth.uid()::text = user_id);

-- Comments: public read, authenticated insert own, update/delete own
alter table comments enable row level security;
create policy "Public read" on comments for select using (true);
create policy "Auth insert" on comments for insert
  with check (auth.uid()::text = user_id);
create policy "Auth update own" on comments for update
  using (auth.uid()::text = user_id);
```

### Why this shape

- **Denormalized scores on `protocols`**: Avoids a join for the browse/card view where you want to show novelty/feasibility/impact badges. The ideation process data is always 1:1 with the protocol.
- **Normalized topics**: Topics are reused across protocols. A `protocol_topics` junction table enables faceted filtering (`WHERE topic_id = 'glp1-receptor-agonists'`). The `topics` table is tiny (tens of rows) and can be loaded once for filter dropdowns.
- **`protocol_references` as a table**: Enables cross-protocol reference search ("show me all protocols that cite Perlas et al.") and role-based filtering.
- **`protocol_files` as a table**: Decouples storage location from the protocol record. The hydration script uploads files and records paths. The frontend fetches from Supabase Storage using the `storage_path`.
- **Clerk user IDs as text**: Clerk's `userId` is a string. No foreign key to a local users table — Clerk is the source of truth for user identity. If you want to display usernames, fetch from Clerk's API or use their `<UserButton>` / `useUser()` components.

## 3. Hydration Script

A Python script (`scripts/hydrate_supabase.py`) that:

1. Scans `workspace/runs/*/` for directories containing `output/protocol_metadata.json`
2. For each run:
   - Reads the metadata JSON
   - Upserts into `protocols`, `topics`, `protocol_topics`, `protocol_references`, `study_designs`
   - Uploads each workspace markdown file to Supabase Storage at `protocols/{protocol_id}/{logical_name}.md`
   - Upserts into `protocol_files` with the storage path
3. Skips runs already in the database (by `run_id`) unless `--force` is passed

This is a ~150 line Python script using `supabase-py`. Run it manually after each batch of agent runs, or after running `extract_metadata.py` on new runs.

### Workflow

```bash
# 1. Run the agent
python main.py ideation

# 2. Extract metadata (post-processing)
python scripts/extract_metadata.py workspace/runs/default/20260409_...

# 3. Hydrate Supabase
python scripts/hydrate_supabase.py workspace/runs/default/20260409_...
# or hydrate all:
python scripts/hydrate_supabase.py --all
```

## 4. Frontend Architecture (Next.js)

### Tech stack

| Layer | Choice | Rationale |
|---|---|---|
| Framework | **Next.js 15 (App Router)** | Server components for SEO + fast initial load; server actions for mutations |
| Auth | **Clerk** | Drop-in `<SignIn>`, `<UserButton>`, middleware protection; user IDs for votes/comments |
| Database | **Supabase** (Postgres) | Full-text search, RLS, storage, real-time (optional for live comment updates) |
| Storage | **Supabase Storage** | Markdown files served as text; no CDN complexity |
| Styling | **Tailwind CSS + shadcn/ui** | Fast to build, consistent, accessible components |
| Markdown | **react-markdown + remark-gfm + rehype-slug** | Tables, GFM, heading anchors for TOC |
| Search | **Supabase `to_tsvector` + `websearch_to_tsquery`** | Postgres full-text search is more than sufficient at this scale |

### Pages / Routes

```
/                           → Browse all protocols (cards grid)
/protocols/[id]             → Protocol detail (tabbed document viewer)
/protocols/[id]/[file]      → Deep link to a specific document tab
/search?q=...&specialty=... → Search results with faceted filters
```

### Browse page (`/`)

- Grid of protocol cards, each showing:
  - `short_title` as badge, `title` as heading
  - `abstract` (truncated to 2 lines)
  - `specialty` + `study_type` as colored badges
  - Top 3-4 topic pills
  - Novelty / Feasibility / Impact scores as small gauges or numbers
  - Vote count (sum of `value` from `votes`)
  - Comment count
- Filter sidebar:
  - Specialty dropdown (from controlled vocabulary)
  - Study type dropdown (from controlled vocabulary)
  - Topic multi-select (from `topics` table)
  - Score sliders (novelty ≥ X, impact ≥ X)
- Sort: newest first, highest impact, most votes
- Pagination or infinite scroll

### Protocol detail page (`/protocols/[id]`)

**Header section:**
- Full title, short title badge
- Abstract
- Metadata badges: specialty, study type, multicenter/randomized indicators
- Score display: three labeled scores (novelty, feasibility, impact) as visual indicators
- Upvote/downvote buttons (Clerk-gated — show login prompt if unauthenticated)
- Net vote count

**Tabbed document viewer:**
| Tab | Source | Notes |
|---|---|---|
| Protocol | `output/study_protocol.md` | Default tab; auto-generated TOC from `##` headings; sticky sidebar nav |
| Landscape Report | `survey/landscape_report.md` | The literature survey |
| Candidate Ideas | `ideas/candidate_ideas.md` | All 5-10 ideas the agent generated |
| Critique Report | `critiques/critique_report.md` | Scores, prior art findings, verdicts |
| Principal Notes | `scratchpad/principal_notes.md` | Decision rationale |

Each tab fetches the markdown from Supabase Storage and renders with `react-markdown`. The protocol tab gets a generated table of contents from `##` headings.

**References sidebar (on protocol tab):**
- List of key references from `protocol_references`
- Each shows: authors, year, journal, title
- Clickable DOI → `https://doi.org/{doi}` (opens in new tab)
- Clickable PMID → `https://pubmed.ncbi.nlm.nih.gov/{pmid}/`
- Color-coded by `role` (background = gray, gap-evidence = amber, prior-art = blue, methodology = green)

**Comments section (below document viewer):**
- Threaded comments (flat for v1, threaded in v2)
- Clerk-gated: unauthenticated users see comments but get a "Sign in to comment" prompt
- Each comment shows user avatar (from Clerk), name, timestamp, body
- Edit/delete own comments

**Study design details (collapsible panel or sidebar):**
- Multicenter: yes/no
- Control arm: yes/no
- Randomization: yes/no
- Duration: N months
- Sample size: N
- Budget: $X
- Data sources: list
- Primary outcome
- Secondary outcomes

### Search page (`/search`)

- Search bar with debounced query
- Supabase full-text search: `protocols.fts @@ websearch_to_tsquery('english', $query)`
- Also searches `keywords` array and topic labels
- Faceted filters same as browse page
- Results rendered as the same card component

## 5. Recommended Metadata Additions

Before building the frontend, I'd recommend these additions to `extract_metadata.py`:

### Add `rejected_ideas` to schema

```python
class RejectedIdea(BaseModel):
    title: str
    reason: str = Field(description="One sentence on why it was not selected.")
    scores: IdeationScores

class ProtocolMetadata(BaseModel):
    # ... existing fields ...
    rejected_ideas: list[RejectedIdea] = Field(
        description="Ideas the agent evaluated but did not select for the final protocol."
    )
```

This data is rich in the critique report (scores, verdicts, reasoning for each of the 6-8 ideas). It adds an "Alternatives Considered" section to the detail page — great UX that demonstrates the depth of the ideation process.

### Add display-friendly names for files

The `files` field currently maps logical names to paths. The frontend needs human-readable labels. Either add a `display_name` mapping in the hydration script, or add it to the metadata:

```python
FILE_DISPLAY_NAMES = {
    "protocol": "Study Protocol",
    "landscape_report": "Landscape Report",
    "candidate_ideas": "Candidate Ideas",
    "critique_report": "Critique Report",
    "novelty_checks": "Novelty Checks",
    "principal_notes": "Principal Notes",
    "critic_notes": "Critic Notes",
    "raw_sources": "Raw Sources",
}
```

This is trivial — keep it in the hydration script or as a frontend constant.

## 7. Open Questions

1. **Real-time comments?** Supabase Realtime can push new comments to all viewers. Nice but not essential for v1.
2. **Reference enrichment?** Could fetch abstracts from PubMed API (via PMID) or Crossref (via DOI) to show richer reference cards. v2 enhancement.
3. **Multi-protocol reference graph?** Once there are 10+ protocols, a "papers cited across protocols" view could be interesting. v2.
4. **Protocol comparison?** Side-by-side comparison of two protocols. v2.
5. **Markdown content search?** Full-text search currently hits metadata fields. Searching within the protocol body text would require storing it in the database (not just Storage). Could add a `body_text` column to `protocols` for FTS indexing, or use Supabase Storage search if available.
6. **Batch hydration automation?** Could add a post-run hook in `runtime/runner.py` that auto-runs `extract_metadata.py` after the ideation agent completes. Then a cron or webhook could trigger `hydrate_supabase.py`. Not needed for v1 — manual is fine.
